## 2. 构建 Verilator NoC 模型

### 2.1 本地构建 NoC

```bash
./scripts/build_noc.sh
```

这个脚本会完成：

- Chisel 生成 RTL
- 自动读取 `generated/openbpu_noc_meta.env`
- 生成 `noc_top.sv` shim
- 调用 Verilator
- 生成 `verilator-noc-lib/build/obj_dir/Vnoc_top__ALL.a`

### 2.2 跳过 Chisel，只重用已生成 RTL

```bash
NOC_SKIP_CHISEL=1 ./scripts/build_noc.sh
```

### 2.3 运行独立 request-path probe

用于在不启动 GPGPU-Sim 的情况下，直接测试 Verilator request 网络：

```bash
NOC_MODE=single ACTIVE_SOURCES=1 PACKETS_PER_SOURCE=1 MAX_CYCLES=128 \
./scripts/run_noc_probe.sh
```

常用模式：

```bash
# 单包最小复现
NOC_MODE=single ACTIVE_SOURCES=1 PACKETS_PER_SOURCE=1 MAX_CYCLES=128 \
./scripts/run_noc_probe.sh

# 多源 round-robin
NOC_MODE=roundrobin ACTIVE_SOURCES=4 PACKETS_PER_SOURCE=1 MAX_CYCLES=256 \
./scripts/run_noc_probe.sh

# 多源热点竞争
NOC_MODE=hotspot ACTIVE_SOURCES=4 PACKETS_PER_SOURCE=4 MAX_CYCLES=512 \
./scripts/run_noc_probe.sh
```

### 2.4 运行热点 sweep

用于快速观察不同 source 数和 burst 深度下的 request-path 退化边界：

```bash
NOC_MODE=hotspot \
ACTIVE_SOURCES_LIST='1 2 4' \
PACKETS_PER_SOURCE_LIST='1 2 4' \
MAX_CYCLES=512 \
./scripts/run_noc_probe_sweep.sh
```

输出列含义：

- `sources`：并发源数
- `pps`：每个源注入的 packet 数
- `injected`：总注入包数
- `delivered`：总送达包数
- `undelivered`：未送达包数
- `throughput`：包/周期吞吐

probe 输出的关键字段包括：

- `injected`
- `delivered`
- `undelivered`
- `avg_latency_cycles`
- `throughput_packets_per_cycle`
- `held_input_cycles`
- `duplicate_output_flits_suppressed`
- `repeated_output_cycles_suppressed`
- `delivered_dst[...]`

补充说明：

- `held_input_cycles` 反映 wrapper 为兼容当前 RTL 而把同一输入 flit 持续呈现了多少拍。
- `duplicate_output_flits_suppressed` 反映 wrapper 在输出侧去重了多少重复 packet。
- 如果去重计数不为 `0`，说明 probe 的“全送达”结果中带有 wrapper 补偿效应，不能直接等同于纯 RTL 已经无误。

### 2.5 运行 input-hold sweep

用于快速比较不同 `MIN_INPUT_HOLD_CYCLES` 对 request-path 送达率的影响：

```bash
HOLDS='1 2 3 4 5 6' \
NOC_MODE=hotspot \
ACTIVE_SOURCES=4 \
PACKETS_PER_SOURCE=4 \
MAX_CYCLES=512 \
./scripts/run_noc_hold_sweep.sh
```

当前本地观测里：

- `hold=1` 时几乎完全不出包
- `hold=3` 时 `4x4 hotspot` 大约为 `16 injected / 8 delivered`
- 当前脚本默认 `hold=5`，同一场景可提升到 `16 injected / 9 delivered`
- `hold=6` 又会回落

补充说明：

- 服务器当前的 Verilator `4.204` 不是按 `VL_THREADED` 方式发布的，因此
  `scripts/run_noc_probe.sh` 已调整为不再强制编译
  `verilated_threads.cpp`。
- 这样 macOS 本地和 Ubuntu 服务器都可以直接复用同一条 probe 命令。

服务器上的一次实际 sweep 结果如下：

```bash
HOLDS='1 3 5' \
NOC_MODE=hotspot \
ACTIVE_SOURCES=4 \
PACKETS_PER_SOURCE=4 \
MAX_CYCLES=512 \
./scripts/run_noc_hold_sweep.sh
```

实测摘要：

- `hold=1`：`16 injected / 0 delivered`
- `hold=3`：`16 injected / 16 delivered`，但 `duplicate_output_flits_suppressed = 27`
- `hold=5`：`16 injected / 16 delivered`，但 `duplicate_output_flits_suppressed = 27`

这说明：

- 远端 standalone probe 在 `hold=3/5` 下可以报“全送达”。
- 但这个结果仍然依赖 wrapper 输出去重补偿，不能直接等同于原始 RTL 已经健康。

## 4. Rodinia 运行命令

### 4.1 运行默认 workload

当前脚本默认 `RODINIA_APP=backprop`：

```bash
./scripts/run_rodinia.sh local
```

### 4.2 显式运行 backprop

```bash
RODINIA_APP=backprop ./scripts/run_rodinia.sh local
```

### 4.3 使用 OpenBPU Verilator backend

`network_mode 3` 表示切换到 OpenBPU backend：

```bash
RODINIA_APP=backprop \
GPGPUSIM_NETWORK_MODE=3 \
./scripts/run_rodinia.sh local
```

如果希望显式覆盖 request-side hold 拍数，可追加：

```bash
RODINIA_APP=backprop \
GPGPUSIM_NETWORK_MODE=3 \
OPENBPU_CONFIG_EXTRA='-openbpu_req_min_input_hold_cycles 5' \
./scripts/run_rodinia.sh local
```

### 4.4 服务器上建议使用的 bounded backprop 命令

```bash
cd /home/hujian/openbpu3-noc
export CUDA_INSTALL_PATH=/usr/local/cuda-11.8
export PATH=/usr/local/cuda-11.8/bin:/usr/local/bin:$PATH
export LD_LIBRARY_PATH=/usr/local/cuda-11.8/lib64:$LD_LIBRARY_PATH

RODINIA_APP=backprop \
GPGPUSIM_NETWORK_MODE=3 \
GPGPUSIM_DEADLOCK_DETECT=0 \
GPGPUSIM_MAX_CYCLE=20000 \
OPENBPU_CONFIG_EXTRA='-openbpu_req_min_input_hold_cycles 3' \
./scripts/run_rodinia.sh local
```

或：

```bash
RODINIA_APP=backprop \
GPGPUSIM_NETWORK_MODE=3 \
GPGPUSIM_DEADLOCK_DETECT=0 \
GPGPUSIM_MAX_CYCLE=20000 \
OPENBPU_CONFIG_EXTRA='-openbpu_req_min_input_hold_cycles 5' \
./scripts/run_rodinia.sh local
```

### 4.5 服务器实测结果摘要

在 `2026-04-28` 的远端实测中，`hold=3` 与 `hold=5` 两组 bounded
`backprop` 结果基本一致：

- `gpu_sim_cycle = 20000`
- `gpu_sim_insn = 1678208`
- `Req_Network__packets_injected = 1462`
- `Req_Network__packets_delivered = 821`
- `Req_Network__avg_latency_cycles = 6878.2387`
- `Req_Network__throughput_packets_per_cycle = 0.0411`
- `Reply_Network__packets_injected = 812`
- `Reply_Network__packets_delivered = 811`
- `Reply_Network__avg_latency_cycles = 20.0000`
- `Reply_Network__throughput_packets_per_cycle = 0.0406`
- `outstanding_payloads = 642`

结论：

- 当前链路已经可以在服务器上完成 `Chisel -> Verilator -> GPGPU-Sim -> Rodinia`
  的集成运行。
- 但 request path 仍然没有完全 drain，停在 `20000` cycle 时还有大量未完成请求。
- 单纯把 wrapper hold 从 `3` 调到 `5`，暂时没有带来全系统层面的进一步改善。
- standalone probe 与全系统 bounded `backprop` 之间仍然存在明显落差，
  后续需要继续盯 request path 的真实前进性，而不是只看 probe 的“送达率”。

### 6.1 查看运行日志

```bash

```

### 6.2 提取关键 NoC 指标

```bash
grep -E 'Req_Network__packets_(injected|delivered)|Req_Network__avg_latency_cycles|Req_Network__throughput_packets_per_cycle|Reply_Network__packets_(injected|delivered)|Reply_Network__avg_latency_cycles|Reply_Network__throughput_packets_per_cycle|gpgpu_simulation_time|GPGPU-Sim: \*\*\* exit detected \*\*\*' run/backprop_noc_stats.log
```

目前最关键的观测指标包括：

- request 注入包数
- request 送达包数
- request 平均延迟
- request 吞吐
- reply 注入/送达包数
- reply 平均延迟
- 总模拟时间

## 7. 当前推荐调试顺序

如果要继续调试 OpenBPU backend，建议按下面顺序执行：

### 7.1 先验证 Chisel 测试

```bash
sbt test
```

也可以先跑更严格的单拍端到端测试：

```bash
sbt "testOnly *SinglePulseEndToEndSpec"
sbt "testOnly *MultiSourceHotspotSpec"
```

其中：

- `SinglePulseEndToEndSpec` 用于验证单拍输入脉冲不会被静默丢失。
- `MultiSourceHotspotSpec` 当前包含两个 `pendingUntilFixed` 诊断用例：
  - 自然一拍 `2x2` 热点突发会掉包
  - 为贴近 wrapper 而持续拉高 `valid` 的 `2x2` 热点突发会产生重复包

### 7.2 再验证 Verilator 构建

```bash
NOC_FORCE_BUILD_NOC=1 ./scripts/build_sim.sh local
```

### 7.3 最后跑 bounded Rodinia

```bash
RODINIA_APP=backprop \
GPGPUSIM_NETWORK_MODE=3 \
GPGPUSIM_DEADLOCK_DETECT=0 \
GPGPUSIM_MAX_CYCLE=20000 \
./scripts/run_rodinia.sh local
```
